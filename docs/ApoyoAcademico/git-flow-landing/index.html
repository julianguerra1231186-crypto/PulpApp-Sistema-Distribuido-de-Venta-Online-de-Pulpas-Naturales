Proceso

Flujo de Trabajo

6 pasos para crear y mergear una Historia de Usuario correctamente

01

Posicionarse en rama padre

Ubícate en la rama desde donde nacerá tu feature.

git switch {ambiente}
02

Crear rama de la HU

Crea y cambia a una nueva rama basada en el padre.

git switch -c HU-01-{ambiente}
03

Desarrollar la HU

Realiza commits respetando Conventional Commits.

git commit -m "feat(auth): add login validation"
04

Sincronizar con padre

Trae cambios recientes del padre antes del MR.

git pull origin {ambiente}
05

Publicar cambios

Sube tu rama al repositorio remoto.

git push origin HU-01-{ambiente}
06

Crear Merge Request

Crea MR hacia la rama padre. Nunca push directo.

GitLab/GitHub → New MR

Ambiente de Desarrollo

develop

¿Por qué existe? Es el punto de integración continua del equipo. Aquí todos los desarrolladores fusionan sus Historias de Usuario mientras trabajan activamente. Permite detectar conflictos de integración de forma temprana, antes de que el código avance a entornos más estables.

Reglas clave

  • Nunca se hace commit directo — solo via Merge Request
  • Es la rama padre de todas las HU de desarrollo
  • El código aquí puede estar incompleto o en prueba
  • Siempre se actualiza antes de crear una nueva HU

Paso a paso en develop

  1. 1
    Posicionarse en develop git switch develop
  2. 2
    Actualizar desde remoto git pull origin develop
  3. 3
    Crear rama de la HU git switch -c HU-01-develop
  4. 4
    Desarrollar con commits convencionales git commit -m "feat(scope): description"
  5. 5
    Sincronizar y publicar git pull origin develop && git push origin HU-01-develop
  6. 6
    Abrir Merge Request → develop GitLab / GitHub → New MR
Ambiente de Pruebas

qa

¿Por qué existe? QA (Quality Assurance) es el filtro de calidad antes de producción. Aquí el equipo de pruebas o el propio desarrollador valida que las funcionalidades cumplan los criterios de aceptación definidos en cada HU. Un bug encontrado aquí cuesta mucho menos que uno en producción.

Reglas clave

  • Solo entra código que ya fue aprobado en develop
  • Cada HU genera su propia rama: HU-XX-qa
  • Si falla la validación, se regresa a develop para corregir
  • Nunca se hace push directo — siempre vía MR

Paso a paso en qa

  1. 1
    Posicionarse en qa git switch qa
  2. 2
    Actualizar desde remoto git pull origin qa
  3. 3
    Crear rama de la HU para qa git switch -c HU-01-qa
  4. 4
    Aplicar ajustes/correcciones de QA git commit -m "fix(scope): resolve qa finding"
  5. 5
    Sincronizar y publicar git pull origin qa && git push origin HU-01-qa
  6. 6
    Abrir Merge Request → qa GitLab / GitHub → New MR
Acumulador de Versiones

release.X

¿Por qué existe? Release actúa como el "empaque final" de un corte académico o sprint. Agrupa todas las HU que pasaron QA en una versión coherente y lista para producción. Permite etiquetar versiones (release.1, release.2) y mantener un historial claro de qué se entregó en cada corte.

Reglas clave

  • Solo entra código validado en qa
  • Se versiona por corte: release.1, release.2, release.3...
  • Acumula todas las HU del corte antes de ir a main
  • Solo admite hotfixes menores; no features nuevas

Paso a paso en release.X

  1. 1
    Posicionarse en release.X git switch release.1
  2. 2
    Actualizar desde remoto git pull origin release.1
  3. 3
    Crear rama de la HU para release git switch -c HU-01-release
  4. 4
    Commits de estabilización / release notes git commit -m "chore(release): prepare v1.0 changelog"
  5. 5
    Publicar la rama git push origin HU-01-release
  6. 6
    MR → release.X (acumula el corte) Todos los MR del corte → release.X
Producción

main

¿Por qué existe? Main es la rama sagrada: representa el código que está (o estará) en producción real. Todo lo que llega aquí ha superado develop, QA y release. Un error en main afecta a todos los usuarios; por eso esta rama es la más protegida y solo recibe MR desde release.

Reglas clave

  • Jamás se hace push o commit directo
  • Solo recibe MR provenientes de release.X
  • Cada MR a main representa un corte/versión completa
  • Se etiqueta con un tag: git tag v1.0

Paso a paso para llegar a main

  1. 1
    Verificar que release.X está completo y probado git log release.1 --oneline
  2. 2
    Abrir MR: release.X → main Un solo MR por todo el corte
  3. 3
    Revisión de código obligatoria (Code Review) Aprobación del líder técnico
  4. 4
    Merge a main aprobado Merge commit: release.1 → main
  5. 5
    Etiquetar la versión git tag v1.0 && git push origin v1.0
  6. 6
    Despliegue a producción Deploy listo
Importante

Reglas de Oro

5 reglas fundamentales que todo el equipo debe seguir

Nunca commits directos

No alterar directamente develop, qa, release ni main. Siempre usar MR.

Siempre git pull antes de MR

Obtener los cambios de la rama padre antes de crear el Merge Request.

Conventional Commits

Usar formato: feat(), fix(), docs(), refactor(), test(), chore()

HU nacen del padre

Las ramas de HU siempre se crean desde su rama padre correspondiente.

Release acumula cortes

Únicamente la rama release se llena de los cambios del corte antes de ir a main.

Comandos Git

Comandos esenciales para el flujo de trabajo

Navegación de Ramas

git switch develop Cambiar a rama develop
git switch -c HU-01-develop Crear y cambiar a nueva rama

Conventional Commits

git commit -m "feat(auth): add JWT validation" Nueva funcionalidad
git commit -m "fix(api): resolve null pointer" Corrección de bug
git commit -m "docs(readme): update installation" Documentación

Sincronización

git pull origin develop Traer cambios del padre
git push origin HU-01-develop Subir rama al remoto

Utilidades

git branch -a Ver todas las ramas
git log --oneline -10 Ver últimos 10 commits
git status Estado actual del repositorio
